# CLAUDE.md

## Project overview

**rebatch** is a CLI tool for sending bulk emails via Resend's Marketing/Broadcast API, using Supabase as a tracking database. It works around Resend's free-tier limits (100 transactional emails/day, audiences capped at 1000 contacts) by batching: adding ~100 contacts to a segment, sending a broadcast, removing them, and repeating.

The tool is resumable — interrupted sends can be re-run and will pick up where they left off (only `pending`/`failed` recipients are processed).

## Architecture

```
src/
├── index.js              # CLI entry point (commander) — commands: init, import, send, status, delete, config
├── init.js               # Interactive setup: config generation + Supabase schema + Edge Function deploy
├── edge-function.js      # Generates the unsubscribe Edge Function TypeScript source
├── broadcaster.js        # Core batch loop (6-step: add → create → send → poll → remove → update)
├── resend-client.js      # Resend API wrapper with rate limiting + retry (max 5 retries, exponential backoff)
├── supabase-client.js    # Supabase helpers — import, query pending, update status, group stats
├── csv-reader.js         # CSV parsing (csv-parse) + email validation + dedup + batching
└── logger.js             # Console output: colors, progress bars, step display
```

**Data flow:** CSV → Supabase (import) → Resend Broadcast (send) → Supabase status update

## Key concepts

- **Groups**: Named labels passed as CLI arguments to organize contacts in Supabase. CSV path and template name are provided via mandatory CLI flags (`--csv`, `--template`).
- **Batching**: Contacts are split into batches (default 100). Each batch goes through a 6-step cycle against the Resend API.
- **Segment protection**: Before processing, the broadcaster snapshots existing segment contacts and only removes contacts it added.
- **Statuses**: `pending`, `failed`, `sent`, `already_contacted`, `unsubscribed`.

## Tech stack

- **Runtime**: Node.js 18+ (uses native `fetch`)
- **Dependencies**: `@supabase/supabase-js`, `commander`, `csv-parse`, `jszip`
- **APIs**: Resend (broadcasts, contacts, templates), Supabase (PostgreSQL + Management API + Edge Functions)
- **Module system**: ESM (`"type": "module"`)
- **CLI binary**: `rebatch` (via `bin` in package.json)

## Commands

```bash
rebatch init                                        # Interactive setup
rebatch import <group> --csv <path> [--dry-run]     # CSV → Supabase
rebatch send <group> --template <name> [--dry-run]  # Supabase → Resend Broadcast
rebatch status <group>                              # Show group stats
rebatch delete <group>                              # Delete group recipients from Supabase
rebatch config                                      # Show config file path
```

## Configuration

Config is stored in the OS app data directory (generated by `rebatch init`):
- **Windows**: `%APPDATA%/rebatch/config.json`
- **macOS**: `~/Library/Application Support/rebatch/config.json`
- **Linux**: `~/.config/rebatch/config.json`

Config fields:
- `resendApiKey`, `segmentId` — Resend credentials
- `supabaseAccessToken` — Supabase personal access token (used during init for Management API)
- `supabaseUrl`, `supabaseServiceKey` — Supabase project credentials (auto-fetched during init)
- `fromEmail`, `replyTo` — sender info
- `batchSize`, `rateLimitDelayMs`, `pollIntervalMs`, `pollTimeoutMs` — tuning
- `csvColumns` — maps CSV headers to internal fields (email, firstName, lastName, organization)

## Development notes

- Resend rate limit: 2 req/s. The `rateLimitDelayMs` (default 600ms) adds delay before each API call. 429s are retried with backoff.
- CSV reader auto-detects comma/semicolon delimiters and handles BOM.
- Logger auto-detects Windows vs Unix for terminal symbols.
- The `init` command auto-creates the `email_recipients` table in Supabase if it doesn't exist.
